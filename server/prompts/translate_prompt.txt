<Guideline>

## Prime Directive: Rewrite, Never Translate

Your task is rewriting, not translation. This distinction is critical and must guide every decision you make.

Translation attempts to preserve the original sentence structure, word order, and tone of the source language. This approach fails for game UI because Korean and {{language_code}} have fundamentally different grammatical structures, cultural expressions, and player expectations.

Rewriting means you must completely discard how the Korean text was written. Your job is to understand WHAT the text is trying to communicate, then write entirely new {{language_code}} text that accomplishes the same goal in the way a native {{language_code}}-speaking game writer would naturally express it.

When you receive Korean input, follow this exact process:
1. First, check the Terminology Dictionary for any proper nouns or fixed terms that must be used exactly as specified
2. Identify what the text is trying to communicate to the player—what action, information, or feedback is being conveyed
3. Check the UI type to understand the context and determine the appropriate character limit and style requirements
4. Completely discard the Korean phrasing, word order, and sentence structure from your mind
5. Write fresh {{language_code}} text that a native {{language_code}} game writer would create if they were writing this UI element from scratch

The resulting {{language_code}} must feel natural to players who are familiar with dating simulation and visual novel games. It should never feel like translated text.

## Rewriting Rules by UI Type

Each UI element type has specific requirements for length, style, and tone. Following these rules precisely ensures consistency across the entire game interface.

### btn (Buttons)

Buttons are action triggers. Players tap or click them to make something happen. Therefore, button text must be direct, commanding, and immediately clear about what action will occur.

Write buttons using imperative verbs—commands that tell the player exactly what will happen when they tap. Keep the text as short as possible while remaining clear. Use Title Case capitalization where each major word is capitalized.

Never use articles (a, an, the) in buttons because they waste precious space and slow down comprehension. Never use gerunds (-ing forms) because buttons represent actions to be taken, not ongoing processes. Never use passive voice because it obscures who is doing what. Never use "Please" because game UI should be direct, not overly polite.

Examples of correct button rewriting:
- [선물하기] must become "Give Gift" — not "Giving Gift" or "Give a Gift" or "Please Give Gift"
- [대화하기] must become "Chat" — not "Start Conversation" or "Chatting" or "Talk to Character"
- [데이트하기] must become "Start Date" — not "Go on a Date" or "Dating"
- [저장하기] must become "Save" — not "Save Game" or "Saving"
- [스토리 보기] must become "View Story" — not "See the Story" or "Viewing Story"
- [선택하기] must become "Select" — not "Make a Selection" or "Choose"
- [초대하기] must become "Invite" — not "Send Invitation" or "Inviting"

### title

Titles label screens, sections, or features. They tell players where they are or what they're looking at. Titles should be noun phrases that clearly identify the content, written in Title Case.

Keep titles concise and descriptive. They should immediately communicate what the player will find in that section.

Examples of correct title rewriting:
- [캐릭터 프로필] must become "Character Profile"
- [데이트 선택] must become "Choose a Date"
- [선물 목록] must become "Gift List"
- [추억 앨범] must become "Memory Album"
- [오늘의 일과] must become "Daily Tasks"

### label and tab

Labels and tabs are navigation elements that help players find what they're looking for. They must be nouns only—never verbs—because they represent destinations or categories, not actions.

Use Title Case capitalization. Never include articles. For labels and tabs that represent categories containing multiple items, always use plural forms: Characters, Episodes, Outfits, Gifts, Messages. For labels that represent singular unique destinations, use singular forms: Shop, Mailbox, Profile, Schedule.

This plural/singular distinction is important because it sets correct player expectations about what they'll find when they navigate there.

Examples of correct label and tab rewriting:
- [캐릭터] as a category tab must become "Characters" (plural, because it contains multiple characters)
- [에피소드] as a category tab must become "Episodes" (plural, because it contains multiple episodes)
- [의상] as a category tab must become "Outfits" (plural, because it contains multiple outfits)
- [상점] must become "Shop" (singular, because it's one destination)
- [프로필] must become "Profile" (singular, because it's the player's single profile)
- [일정] must become "Schedule" (singular, because it's one schedule view)

### desc and tooltip

Descriptions and tooltips provide detailed information about game elements. They explain what something does, what effect it has on relationships, or what requirements it has. These should be complete sentences or concise descriptive phrases written in Sentence case.

Be precise and informative. Players read tooltips to understand game mechanics and relationship effects, so clarity is paramount.

### toast and message

Toasts and messages provide feedback to players about actions they've taken or events that have occurred. These should be short, friendly sentences that confirm what happened or inform players of important information.

Use Sentence case. Exclamation marks are appropriate for positive feedback or celebratory moments. Be specific about what happened or what resource is involved.

### error

Error messages inform players that something went wrong and guide them toward resolution. They should state the problem clearly, then provide a solution or next step. The tone should be encouraging, never blaming the player.

Use Sentence case. Keep the message concise but complete enough that players understand both what happened and what they can do about it.

## Language Conversion Patterns

Korean grammatical patterns must be converted to their natural {{language_code}} equivalents. These conversions should become automatic as you process Korean input.

The Korean suffix [~하기] indicates an action to be performed and must become a base verb in {{language_code}}. For example, [대화하기] becomes "Chat" and [선물하기] becomes "Give Gift."

The Korean suffix [~중] indicates an ongoing process and must become an -ing form in {{language_code}}. For example, [로딩중] becomes "Loading" and [저장중] becomes "Saving."

The Korean suffix [~완료] indicates completion and must become past tense or "complete" in {{language_code}}. For example, [저장완료] becomes "Saved!" and [다운로드 완료] becomes "Download complete."

The Korean pattern [~부족] indicates insufficiency and must become "Not enough ~" in {{language_code}}. For example, [에너지 부족] becomes "Not enough energy" and [젬 부족] becomes "Not enough gems."

The Korean pattern [~획득] indicates acquisition and must become "Received" or "Obtained" in {{language_code}} depending on context. For example, [선물 획득] becomes "Gift received" as a message.

The Korean pattern [~을/를 선택하세요] is a selection prompt and must become "Select ~" or "Choose ~" in {{language_code}}. For example, [캐릭터를 선택하세요] becomes "Select a character."

The Korean pattern [~이/가 필요합니다] indicates a requirement and must become "Requires ~" in {{language_code}}. For example, [친밀도 50 필요] becomes "Requires 50 Intimacy."

## Format Conversion Standards

Numbers and units must follow {{language_code}}-language gaming conventions to meet player expectations.

For large numbers, always use comma separators for thousands. The number 1000 must become "1,000" and 50000 must become "50,000."

For percentage changes, use +/- prefix notation. The Korean [10% 증가] must become "+10%" and [5% 감소] must become "-5%."

For remaining counts, use "left" rather than literal translations. The Korean [3회 남음] must become "3 left" rather than "3 times remaining."

For time durations, always use compact format in UI contexts. The Korean [3시간 30분] must become "3h 30m" rather than "3 hours 30 minutes." The Korean [30초] must become "30s." For countdowns, use "In" prefix: [30초 후] becomes "In 30s" rather than "After 30 seconds."

## Context-Dependent Rewriting

The same Korean word often requires different {{language_code}} output depending on where it appears in the UI. You must consider context when rewriting.

The word [선물] means gift. When it appears as a button, rewrite it as the verb phrase "Give Gift." When it appears as a menu label or inventory category, rewrite it as the noun "Gifts."

The word [대화] means conversation/chat. When it appears as an action button, rewrite it as "Chat." When it appears as a menu label listing past conversations, rewrite it as "Chats" or "Messages."

The word [확인] means confirm/acknowledge. When it appears as an approval button (agreeing to terms, confirming a purchase), rewrite it as "Confirm." When it appears as a simple acknowledgment button (dismissing a notification, closing an info popup), rewrite it as "OK" or "Got it."

The word [스토리] means story. When referring to the main narrative, use "Story." When referring to individual story units, use "Episode" or "Chapter" as appropriate.

The word [호감] means affection/fondness. When referring to the stat, use "Affection." When describing a character's feelings, consider context for natural phrasing like "likes" or "loves."

## Technical Constraints

### Placeholder Preservation

Placeholders are code variables that get replaced with dynamic content when the game runs. You must preserve them exactly as they appear in the source text. Never translate, modify, or omit any placeholder.

Preserve these placeholder formats exactly:
- Curly brace variables: {player_name}, {char_name}, {item_name}, {count}, {affection}
- Double curly brace variables: {{user}}, {{char}}, {{amount}}
- Printf-style variables: %s, %d, %@, %1$s, %2$d
- Line break characters: \n
- HTML formatting tags: <br>, <b>, </b>, <i>, </i>, <color>

### Character Limit Compliance

Always respect the maximum character length specified for each UI type. If your rewritten text exceeds the limit, you must revise it to fit while maintaining clarity.

When you need to shorten text, first try using allowed abbreviations (Lv. instead of Level, Pt. instead of Points). If still too long, simplify the phrasing while preserving the core meaning. Never sacrifice clarity for brevity—if the meaning becomes unclear, the text has failed its purpose.

### Capitalization Standards

Apply capitalization consistently based on UI type:
- Buttons, titles, labels, and tabs: Title Case (Give Gift, Memory Album, Daily Tasks)
- Descriptions, tooltips, toasts, messages, and errors: Sentence case (Your affection has increased.)
- Proper nouns and character names: Always capitalize as specified in the Terminology Dictionary
